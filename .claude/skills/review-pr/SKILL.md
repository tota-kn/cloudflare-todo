---
name: review-pr
description: GitHubのPRをレビューする。PR番号を省略するとオープンなPR一覧から選択できる。例: /review-pr または /review-pr 2
---

以下の手順でPRレビューを実施してください。

## ステップ1: レビュー対象PRの特定

引数: `$ARGUMENTS`

- **引数が指定されている場合**: PR #$ARGUMENTS をレビュー対象とする
- **引数が空の場合**:
  1. `gh pr list --json number,title,author,headRefName,isDraft` を実行してオープンなPR一覧を取得する
  2. AskUserQuestion ツールを使い、取得したPR一覧（番号・タイトル・作者・ブランチ名）をオプションとして提示し、ユーザーにレビューするPRを選択させる
  3. 選択されたPRをレビュー対象とする

PR番号が確定したら、ステップ2に進んでください。

## ステップ2: PR情報収集（PR番号確定後に実行）

以下のコマンドで情報を収集してください：

- 概要・URL: `gh pr view <PR番号> --json number,title,url,author,body`
- 変更ファイル一覧: `gh pr diff <PR番号> --name-only`
- 差分: `gh pr diff <PR番号>`
- 既存コメント: `gh pr view <PR番号> --comments`

収集後、詳細なコードレビューを日本語で行い、結果を `REVIEW.md` に書き出してください。

## レビュー観点

以下の観点で順番にチェックしてください。

### 1. アーキテクチャ（オニオンアーキテクチャ）

- 依存方向が正しいか: domain ← application ← infrastructure ← presentation
- 各層の責任が明確か（例: ドメイン層にインフラの詳細が漏れていないか）
- ユースケースで複雑なビジネスロジックを統率しているか

### 2. 型安全性

- `AppType` → `ClientType` の型フローが維持されているか
- Zodバリデーションがプレゼンテーション層で行われているか
- `any` や型アサションの不適切な使用がないか

### 3. セキュリティ・認証

- 認証ミドルウェア（authMiddleware）の適用漏れがないか
- ユーザーが他ユーザーのリソースにアクセスできる脆弱性がないか
- シークレット情報がコードに含まれていないか

### 4. TSDoc

- 公開API・型・インターフェースにTSDocが記載されているか
- 変更・追加したシンボルに説明が付いているか

### 5. テスト

- ユースケース層に単体テストがあるか（必須）
- ドメインエンティティの変更に対するテストが更新されているか
- モック・テストファクトリが適切に更新されているか

### 6. マイグレーション

- SQLマイグレーションファイルの変更が整合しているか
- 既存データへの影響（破壊的変更）がないか
- シードデータが実態と合っているか

### 7. コード品質

- 命名がプロジェクト規約に沿っているか
- 重複コードがないか
- エラーハンドリングが適切か

## 出力フォーマット

必ず以下の形式で出力してください：

```
## PR #[番号] レビュー: [タイトル]

**URL**: [GitHubのPR URL]

### 概要
（PRの目的と変更内容を2〜3文でサマリー）

### 変更点サマリー
（変更ファイルをカテゴリ別に整理して説明）

### 指摘事項

#### 🔴 Critical（マージ前に必ず修正）
（ある場合のみ）

#### 🟡 Warning（修正を強く推奨）
（ある場合のみ）

#### 🔵 Suggestion（任意・改善案）
（ある場合のみ）

### 総評
（全体的な品質評価と一言コメント）
```

指摘事項がない場合は「指摘なし」と明記してください。

レビュー完了後、上記フォーマットの内容を `REVIEW.md` にWrite ツールで書き出してください。
